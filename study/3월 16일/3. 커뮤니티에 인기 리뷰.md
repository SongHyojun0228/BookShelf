***[좋아요순 리뷰]***

**DB View 사용**
1. ["인기순 정렬"의 벽]
- `Reviews Table` : 누가? 어떤 책에? 어떤 설명을? 몇 점을?
- `Likes` : 누가? 어떤 리뷰를?
- `문제` : Reviews Table에 좋아요 개수 X

2. [고민]
- `Reviews Table`에 좋아요 개수(likes_count) 칼럼 추가하기 
    - 작업을 두 번 해야 함 : `Likes Table`에 추가 + 좋아요가 눌릴 때마다 해당 칼럼 + 1 
    - 서버 오류가 하나라도 일어나면, 데이터 불일치 
- `View` 만들기
    - `Reviews Table`과 `Likes Table`을 합쳐서 실시간으로 개수를 세주는 가상 테이블
    - 물리적 칼럼을 추가하지 않아도 되고, 실시간으로 개수를 직접 세기 때문에 데이터가 정확

3. [왜 `View`가 예술인가]
3-1. **Supabase(PostgREST)의 기술적 한계 극복**
- 프론트엔드 코드(JavaScript/JS)에서 supabase.from('Reviews').select(...) 만으로는 **"다른 테이블의 행 개수를 세서 그것으로 정렬해줘"**라는 명령을 내릴 수 없음

3-2. **프론트엔드 코드의 단순화**
- 복잡한 계산 로직(Join, Group By, Count)을 백엔드(DB)가 미리 처리
    - View 미사용 시: 리뷰를 다 가져와서 JS로 개수를 세고 정렬해야 함 (느리고 복잡)
    - View 사용 시: 그냥 from('reviews_with_likes') 한 줄이면 끝!

3-3. **데이터 무결성**
- 좋아요가 추가되거나 취소될 때 따로 숫자를 맞춰주는 복잡한 로직(Trigger/Transaction)을 짤 필요가 없습니다. 뷰는 실시간으로 `Likes Table`의 행 개수를 바라보기 때문에 절대 숫자가 틀리지 않음

4. 정리
"물리적인 테이블 구조는 깔끔하게 유지(정규화)하면서, 조회(Select)를 아주 쉽고 강력하게 하기 위해 뷰를 사용했습니다. 덕분에 데이터 불일치 걱정 없이 인기순/최신순 정렬 기능을 완벽하게 구현할 수 있었습니다."

***'Reviews_with_likes'라는 이름의 가상 테이블(View)을 만듭니다.***
CREATE OR REPLACE VIEW public.reviews_with_likes AS
SELECT 
    r.*,                          -- Reviews 테이블의 모든 칼럼
    COUNT(l.like_id) as likes_count -- Likes 테이블에서 해당 리뷰의 좋아요 개수를 센 것
FROM 
    public."Reviews" r
LEFT JOIN 
    public."Likes" l ON r.review_id = l.review_id
GROUP BY 
    r.review_id;
