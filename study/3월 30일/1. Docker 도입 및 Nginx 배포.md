# 2026년 3월 30일: Docker 도입 및 Nginx 배포 최적화

오늘 `devops-learning-roadmap`에 맞춰 프로젝트에 Docker를 도입하고, 운영 환경을 고려한 멀티 스테이지 빌드(Nginx 활용)까지 구현

## 1. Docker화 (Dockerization)
### 목차
- `Dockerfile` 작성 (개발용)
- `docker-compose.yml` 구축
- 환경 변수(`.env`) 연동

### 핵심 요약
- **환경 격리**: 개발자마다 다른 Node.js 버전 문제를 해결하고, 어떤 환경에서도 동일하게 실행되도록 '컨테이너화' 함.
    - 어떻게 해결되는가? 
        : Dockerfile으 첫줄 FROM **node:20-alpine**이 핵심
        : 위 명령어가 들어있는 이미지를 빌드하면, 도커는 내 컴퓨터 OS가 무엇이든 상관없이 내부에 Node 20이 설치된 아주 가벼운 리눅스 컴퓨터 한 대를 생성
- **Docker Compose**: 앱 실행에 필요한 다수의 설정(포트 포워딩, 볼륨 매핑 등)을 파일 하나로 관리. `docker-compose up`으로 한 번에 실행 가능.

## 2. 보안 및 환경 변수 관리
- **Vite 규칙**: 클라이언트 측에서 환경 변수를 읽으려면 반드시 `VITE_` 접두사를 붙여야 함 (예: `VITE_SUPABASE_URL`).
- **보안**: `.env` 파일을 `.gitignore`에 등록하여 민감한 키가 GitHub에 유출되는 것을 방지하고, Docker를 통해 안전하게 주입함.

## 3. 배포 최적화: Multi-stage Build & Nginx
### Nginx과 하는 일
1. **Nginx?**
- 정의: 아주 빠르고 가벼운 '웹 서버(Web Server)' 소프트웨어
- Node.js(Vite Dev Server): 음식을 직접 만들면서 배달도 하는 식당 주인 (개발 모드에서는 편리하지만, 바쁠 때는 비효율적)
- Nginx: 요리가 다 된 음식(빌드된 HTML/JS/CSS 파일)을 손님에게 빛의 속도로 전달만 하는 전문 배달원.

2. **하는 일**
- 우리가 npm run build를 하면 dist 폴더에 정적 파일들(HTML, JS, CSS)이 생성
- Nginx는 사용자가 http://...으로 접속했을 때, 이 dist 폴더 안에 있는 파일들을 그냥 읽어다 전달해 주는 역할
- 계산(빌드)은 이미 끝났으므로, Nginx는 파일을 전달하는 것 외에 아무런 자원을 쓰지 않아 매우 빠르고 안정적

### 멀티 스테이지 빌드 방식
1. **Build Stage**: `node:alpine` 이미지를 사용하여 앱을 빌드 (`npm run build`). 여기서 `dist` 폴더가 생성됨.
    - node:alpine?
        - Node.js : 자바스크립트를 실행하는 엔진
        - Alpine(알파인 리눅스) : 도커 세계에서 가장 유명한 초소형 리눅스 운영체제, 일반적인 Ubuntu 이미지(100MB~)에 비해 Alpine은 5MB 정도 
        - 합치면 : 불필요한 기능(GUI, 복잡한 유틸리티 등)을 다 빼고 **"딱 Node.js를 실행하는 데 필요한 최소한의 리눅스만 남긴 이미지"**라는 뜻
2. **Serve Stage**: 빌드된 `dist` 폴더의 정적 파일들만 `nginx:stable-alpine` 이미지로 복사함. 나머지 무거운 `node_modules`와 Node 엔진은 버림.

### 왜 Nginx인가?
- **성능**: 정적 파일(HTML/JS/CSS)을 서빙하는 데는 Node.js 개발 서버보다 Nginx가 훨씬 빠르고 안정적이며 가벼움.
- **표준**: 실제 운영 환경(AWS, GCP 등)에서 프론트엔드 배포의 표준적인 방식임.

## 4. 결과 비교
| 항목 | 개발용 이미지 (bookshelf-app) | 배포용 이미지 (bookshelf-prod) |
| :--- | :--- | :--- |
| **기반 시스템** | Node.js + Dev Server | Nginx (Lightweight) |
| **이미지 전체 용량** | **396 MB** | **92.3 MB** |
| **실제 콘텐츠(파일) 용량** | 96.7 MB | **25.9 MB** |

> **결론**: 멀티 스테이지 빌드를 통해 전체 용량을 **약 1/4**로 줄였으며, 실제 실행 환경은 훨씬 가벼워짐.

## 5. 주요 명령어 메모
```bash
# 개발용 실행
docker-compose up

# 배포용 이미지 직접 빌드 (환경 변수 주입 필요)
docker build --build-arg VITE_SUPABASE_URL=... --build-arg VITE_SUPABASE_ANON_KEY=... -f Dockerfile.prod -t bookshelf-prod .

# 배포용 컨테이너 실행 (8080 포트)
docker run -d -p 8080:80 --name bookshelf-prod-test bookshelf-prod
```
