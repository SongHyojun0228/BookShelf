# 2. 도커 이미지 업데이트 및 컨테이너 관리 (Troubleshooting)

코드를 수정하거나 GitHub Secrets를 바꾼 뒤, 내 로컬 컴퓨터의 컨테이너에 반영하는 **실전 프로세스**를 정리합니다.

---

## 🔥 핵심 3단계 업데이트 콤보
GitHub Actions 빌드가 끝난 뒤(초록 체크 ✅), 터미널에서 순서대로 입력하세요.

### 1단계: 최신 이미지 내려받기 (Pull)
클라우드(Docker Hub)에 올라온 따끈따끈한 새 이미지를 가져옵니다.
```bash
docker pull [이미지명]:latest
```

### 2단계: 기존 컨테이너 강제 삭제 (Remove Force)
실행 중인 컨테이너는 그냥 `rm`으로 지워지지 않습니다. `-f`(force) 옵션을 붙여서 한방에 끄고 지웁니다.
```bash
docker rm -f [컨테이너_이름]
```

### 3단계: 새 이미지로 다시 실행 (Run)
```bash
docker run -d -p 8080:80 --name [컨테이너_이름] [이미지명]:latest
```

---

## 🧹 도커 청소하기 (Cleanup)
실습을 많이 하다 보면 안 쓰는 이미지와 컨테이너가 쌓여서 컴퓨터 용량을 차지합니다. 그럴 때 쓰는 명령어입니다.

### 안 쓰는 이미지 지우기
```bash
docker rmi [이미지_ID]
```

### 모든 안 쓰는 자원 한방에 청소하기 (강력 추천!)
멈춰있는 컨테이너, 이름 없는 이미지 등을 싹 다 지워줍니다.
```bash
docker system prune -a
```

### 0. GitHub Actions: 자동 배포의 흐름 (CI Process)
"내가 `git push`만 했는데, 왜 Docker Hub에 `latest`가 생길까요?"

1. **방아쇠 (Trigger)**: 당신이 `git push`를 하는 순간, GitHub 서버에 숨어있던 우리의 로봇(`deploy.yml`)이 잠에서 깹니다. ⏰
2. **빌드 (Build)**: 로봇은 당신의 코드를 한 줄씩 읽어서 `Dockerfile.prod`대로 붕어빵(이미지)을 구워냅니다. 이때 이미지는 당신의 아이디와 함께 **`latest`**라는 이름표를 달게 됩니다. 🥨
3. **푸시 (Push)**: 로봇은 완성된 붕어빵을 들고 **Docker Hub** 창고로 달려가서 "여기에 최신판(`latest`) 맡겨둡니다!"라고 소리치며 던져놓고 갑니다. 🚛
4. **결과**: 이 모든 과정이 성공하면 GitHub Actions 화면에는 초록색 체크(`✅`)가 뜨고, Docker Hub에는 방금 도착한 따끈따끈한 `latest` 이미지가 저장됩니다!

## 🏗️ 도커 업데이트의 원리 (Behind the Scenes)
"왜 `git push`를 하고 `docker pull`을 하면 코드가 바뀔까요?" 이 마법 같은 과정의 원리를 알아봅시다.

### 1. `latest`는 이름이 아니라 '스티커'입니다.
우리가 빌드할 때 태그를 `latest`로 달면, Docker Hub라는 창고에 **"이게 최신판이야!"**라는 스티커를 붙이는 셈입니다. 
- 새로 빌드를 해서 또 `latest` 태그를 달면, 기존의 `latest` 스티커가 예전 이미지에서 떨어져서 **새로운 이미지에 옮겨 붙습니다.** 🏷️

### 2. `docker pull`의 역할
내 컴퓨터(로컬)는 기본적으로 **"나한테 이미 `latest`가 있는데?"**라고 생각하며 게으름을 피웁니다. 
- 명시적으로 `docker pull` 명령어를 내리면, 내 컴퓨터는 Docker Hub 창고로 달려가서 **"혹시 `latest` 스티커가 다른 이미지로 옮겨 붙었나?"**를 확인합니다. 
- 만약 옮겨 붙었다면(빌드 번호가 달라졌다면), 그때 비로소 새로운 이미지 데이터를 내려받습니다.

### 3. 레이어 전송 (Layered Download) - 도커가 빠른 이유! ⚡
이미지를 통째로 다시 받는 게 아닙니다. 도커는 이미지를 햄버거 층(Layer)처럼 쌓아서 만듭니다.
- 1층(OS), 2층(Node.js), 3층(코드) 중에서 **바뀐 부분(3층 코드)**만 골라서 다운로드합니다. 이 덕분에 업데이트 속도가 엄청나게 빠른 것입니다!

### 4. `docker rm -f`가 필요한 이유
`docker pull`로 새 이미지를 받았다고 해서, **지금 실행 중인 컨테이너가 마법처럼 저절로 바뀌지는 않습니다.**
- 현재 실행 중인 컨테이너는 **"처음 실행될 당시의 이미지"**를 붙잡고 돌아가고 있습니다. 
- 그래서 예전 컨테이너를 **강제로 지우고(`rm -f`)**, 새로 받은 이미지를 가지고 **새 컨테이너를 띄워야(`run`)** 비로소 바뀐 코드가 적용됩니다! 🦾
